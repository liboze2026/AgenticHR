# 招聘助手 — 设计文档

## 1. 项目概述

### 1.1 背景
公司 HR（无技术背景）使用 Boss 直聘接收大量应聘者简历（约 500 份/天），需要从中筛选合适候选人，并协调面试官安排面试。现有流程全靠手动操作，效率低下。

### 1.2 目标
开发一套本地部署的招聘辅助工具，覆盖从简历采集、智能筛选、面试安排到通知触达的完整流程，极大简化 HR 的日常操作。

### 1.3 核心原则
- **易用性**：所有操作不超过 3 次点击，HR 无需技术背景即可使用
- **健壮性**：完善的容错、重试、异常提示机制
- **高可用**：单个模块故障不影响其他功能
- **解耦性**：每个模块可独立运行和测试
- **一键启动**：双击 exe 即可运行全部服务

### 2.1 技术栈

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| 后端 | Python + FastAPI | 生态丰富，AI 库成熟，打包方便 |
| 前端 | Vue 3 + Element Plus | 轻量，中后台组件丰富，适合表单/表格密集场景 |
| 数据库 | SQLite | 单文件，零配置，适合本地部署 |
| Chrome 插件 | Manifest V3 | 在 Boss 直聘页面上直接操作 |
| 备选自动化 | Playwright (Python) | 与后端同技术栈，控制力强 |
| AI | Claude Haiku / 通义千问-Turbo | 性价比高，日成本约 10-20 元 |
| 打包 | PyInstaller | 打包成单个 exe，HR 双击即用 |

### 2.2 核心数据流

1. **简历采集**：Chrome 插件/Playwright 在 Boss 直聘上自动打招呼 → 候选人回复简历 → 自动下载 PDF → 解析入库；同时通过 IMAP 扫描 HR 邮箱获取邮件简历
2. **智能筛选**：HR 配置岗位筛选规则 → 硬性条件本地过滤 → 通过的送 AI 评估打分（可选）→ HR 查看结果人工决定
3. **面试安排**：HR 选中候选人 → 获取面试官空闲时段 → HR 录入候选人可用时间 → 系统匹配 → HR 确认
4. **通知触达**：确认后 → 创建腾讯会议 → 邮件发候选人 → 飞书通知面试官 → 生成消息模板供复制到 Boss 直聘/微信

## 3. Boss 直聘操作流程

### 3.1 完整流程

```
应聘者在 Boss 直聘上"打招呼"
    ↓
插件/Playwright 检测到新的打招呼消息
    ↓
模拟点击回复，发送预设话术
（如"您好，请发送一份PDF简历给我，谢谢"）
    ↓
等待候选人回复简历
    ↓
两个简历来源并行采集：
  ├─ Boss 直聘聊天中：检测到 PDF 附件 → 自动下载
  └─ HR 邮箱中：IMAP 定时检查 → 匹配候选人 → 自动下载
    ↓
PDF 解析 + 页面信息抓取 → 合并入库
```

### 3.2 打招呼话术管理
- HR 可在设置中自定义多条回复话术模板
- 支持按岗位配置不同话术
- 不同候选人随机选用话术，避免重复单一

### 3.3 反检测策略

| 策略 | 具体措施 |
|------|---------|
| 操作节奏 | 每次操作间随机延迟（3-8秒），模拟人工操作速度 |
| 行为模拟 | 模拟鼠标移动轨迹、滚动页面、随机停顿 |
| 频率控制 | 每小时/每天最大操作次数上限，HR 可调节，默认保守值 |
| 随机化 | 话术随机、间隔随机浮动、浏览时长随机 |
| 分散执行 | 分散在工作时间内执行，不集中操作 |
| 指纹保持 | 使用 HR 自己的 Chrome 和登录态，不伪造 User-Agent |

### 3.4 健壮性保障
- 页面加载未完成不操作，等待关键元素出现后再执行
- 操作失败自动跳过当前候选人，记录日志，继续下一个
- 检测到验证码或账号限制提示时，立即暂停所有自动操作，通知 HR 手动处理
- 所有自动操作可一键暂停/恢复
- 操作日志完整记录每一步动作和结果

### 3.5 多方案适配器

所有 Boss 直聘操作通过统一适配器接口封装，可在设置中切换：

| 优先级 | 方案 | 原理 |
|--------|------|------|
| 主力 | Chrome 插件 | 注入 Boss 直聘页面，直接操作 DOM |
| 备选1 | Playwright | 启动独立 Chrome 实例，程序控制页面 |
| 预留 | Android 模拟器 + Appium | 运行手机端 Boss 直聘 App |
| 预留 | RPA 桌面自动化 (PyAutoGUI) | 基于屏幕图像识别，完全不注入代码 |

第一期实现 Chrome 插件 + Playwright 两套，其余预留接口。

## 4. 后端模块设计

### 4.1 模块划分

7 个独立模块，每个模块有自己的 API 路由、数据模型和测试：

| 模块 | 职责 | 独立运行方式 |
|------|------|-------------|
| 简历管理 (resume) | 接收/存储/查询/导入简历数据，PDF 解析 | 可单独启动，用 API 或上传文件测试 |
| 筛选规则 (screening) | 管理岗位、配置硬性条件和软性要求 | 可单独启动，输入简历 JSON 输出筛选结果 |
| AI 评估 (ai_evaluation) | 调用大模型对简历做评分和匹配分析 | 可单独启动，输入简历+岗位要求输出评分 |
| 日程协调 (scheduling) | 对接飞书日历、管理时间、匹配时段 | 可单独启动，mock 日历数据测试匹配 |
| 会议管理 (meeting) | 创建腾讯会议 / 管理手动填入的链接 | 可单独启动，测试会议创建和查询 |
| 通知 (notification) | 邮件发送、飞书消息推送、消息模板生成 | 可单独启动，发测试邮件/飞书消息 |
| 飞书机器人 (feishu_bot) | 接收飞书指令、调用各模块完成操作 | 可单独启动，用飞书消息驱动测试 |

### 4.2 模块间通信规则
- 模块之间只通过内部 API 接口调用，不直接访问对方的数据库表
- 每个模块定义清晰的输入/输出数据结构（Pydantic schema）
- 任何外部服务通过适配器层封装，可替换 mock 实现进行测试

### 4.3 目录结构


## 5. AI 简历筛选

### 5.1 总开关
AI 功能在设置中可开关。关闭后系统只用硬性条件筛选，所有 AI 相关入口隐藏。

### 5.2 两阶段筛选

**第一阶段 — 硬性条件过滤（本地执行，无 API 成本）：**
- 在 SQLite 中查询，根据 HR 配置的条件过滤：学历、工作年限、期望薪资范围、必备技能关键词等
- 不符合的标记为"不符合硬性条件"，附带原因

**第二阶段 — AI 软性评估（调用大模型 API，可选）：**
- 将简历 + 岗位软性要求发给大模型
- 输出结构化结果：匹配度评分（1-100）、优势点（2-3条）、风险点（2-3条）、综合建议（推荐/待定/不推荐）
- HR 看到评分排序列表，最终人工决定

### 5.3 成本估算
- 500 份/天，硬性条件过滤 60-70% 后约 150-200 份进入 AI
- 每份约 2000 token 输入 + 500 token 输出
- 使用 Claude Haiku 或通义千问-Turbo，日成本约 10-20 元
- 支持在设置中切换 AI 提供商和模型

### 5.4 健壮性
- AI 评估采用异步批量处理，不阻塞 HR 操作
- 单条评估失败自动重试 2 次，仍失败标记为"评估失败"供 HR 手动处理
- AI 服务不可用时，硬性条件筛选仍正常工作

## 6. Web 前端

### 6.1 整体布局
左侧导航栏 + 右侧工作区，典型中后台管理界面。

### 6.2 核心页面

| 页面 | 功能 |
|------|------|
| 工作台（首页） | 今日数据概览：新简历数、待筛选数、待安排面试数、今日面试数 |
| 简历库 | 表格展示所有简历，搜索/筛选/排序/查看详情，显示 AI 评分和筛选状态 |
| 岗位管理 | 创建岗位、配置硬性条件表单 + 自然语言软性要求、查看岗位下候选人 |
| 面试安排 | 日历视图 + 列表视图，展示面试官空闲时段，点击安排面试，一键发送通知 |
| 通知记录 | 已发送通知历史（邮件/飞书/模板），查看发送状态和内容 |
| 设置 | 邮箱配置、飞书凭证、腾讯会议凭证、AI 配置、面试官名单、Boss 直聘话术管理 |

### 6.3 易用性设计
- 所有操作不超过 3 次点击
- 关键操作有确认弹窗防误触（如批量淘汰、发送通知）
- 表单带输入校验和帮助提示
- 操作完成后有明确的成功/失败反馈
- 首次使用有引导设置流程（配置邮箱、飞书凭证等）

### 6.4 用户管理
- 多 HR 登录，简单身份区分
- 登录认证即可，无复杂权限体系

## 7. 飞书机器人

### 7.1 与 Web 端功能对等
飞书机器人是完整的第二操作入口，可独立完成全部操作。

### 7.2 核心指令

| 指令 | 功能 |
|------|------|
| 查看概览 | 返回今日数据卡片 |
| 查看简历 | 返回待处理简历列表卡片，支持翻页和查看详情 |
| 筛选简历 | 选择岗位 → 触发筛选 → 返回结果，点击按钮通过/淘汰 |
| 安排面试 | 选候选人 → 显示空闲时段 → 选定 → 确认安排 |
| 发送通知 | 一键触发邮件 + 飞书通知 + 生成消息模板 |
| 管理岗位 | 创建/编辑岗位和筛选规则 |

### 7.3 主动推送

**发给 HR：**
- 新简历采集完成通知
- AI 筛选完成通知
- 今日面试提醒

**发给面试官：**
- 面试安排通知（候选人信息 + 时间 + 会议链接）
- 面试前 30 分钟提醒

### 7.4 交互设计
- 使用飞书消息卡片（Interactive Card），按钮点击直接触发操作
- 复杂操作（如配置筛选规则）引导跳转 Web 端

## 8. 日程协调与会议管理

### 8.1 面试安排流程

```
HR 选中候选人 → 选择面试官
    ↓
获取面试官空闲时段（三种方式优先取可用的）：
  ├─ 优先：读取飞书日历 freebusy
  ├─ 补充：面试官手动填写的可用时间
  └─ 兜底：HR 手动指定
    ↓
HR 录入候选人可用时间
    ↓
系统匹配双方都空闲的时段，列出可选项
    ↓
HR 选定时段，确认安排
    ↓
自动创建腾讯会议（或 HR 手动填入链接）
    ↓
一键触发通知：
  ├─ 邮件 → 候选人（面试时间 + 会议链接 + 注意事项）
  ├─ 飞书消息 → 面试官（候选人信息 + 时间 + 会议链接 + 简历）
  └─ 生成消息模板 → 供复制到 Boss直聘/微信
```

### 8.2 冲突处理
- 安排面试时自动检测面试官是否已有其他面试，有冲突则提示
- 同一候选人不能被安排重叠时段的面试

### 8.3 腾讯会议
- 优先通过腾讯会议 API 自动创建（需在设置中配置凭证）
- API 不可用时退回手动模式：HR 自己创建后填入链接
- 会议信息（链接、密码、时间）随通知一并发出

## 9. 通知渠道

### 9.1 通知对象与渠道

| 对象 | 渠道 | 方式 |
|------|------|------|
| 候选人 | 邮件 | 系统自动发送（主力渠道） |
| 候选人 | Boss 直聘站内消息 | 系统生成模板，HR 复制粘贴 |
| 候选人 | 微信等其他方式 | 系统生成模板，面试官/HR 手动复制发送 |
| 面试官 | 飞书消息 | 系统自动推送 |
| HR | 飞书消息 | 系统自动推送（提醒、状态更新） |

### 9.2 HR 邮箱采集
- 通过 IMAP 协议定时扫描（可配置间隔，默认 5 分钟）
- 根据发件人邮箱或邮件标题关键词匹配候选人
- 自动下载 PDF 附件并与已有候选人记录关联
- 无法匹配的邮件标记为"待人工关联"

## 10. 部署方案

### 10.1 一键启动（HR 日常使用）
- 使用 PyInstaller 打包成单个 `招聘助手.exe`
- 双击启动：自动启动后端服务 + 打开浏览器访问前端
- 系统托盘图标显示运行状态
- 首次运行自动初始化 SQLite 数据库

### 10.2 单模块启动（开发/测试）
- 每个模块可通过命令独立运行，如 `python -m app.modules.resume`
- 支持 mock 外部服务进行隔离测试
